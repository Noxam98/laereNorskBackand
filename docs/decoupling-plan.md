# План декомпозиции ядра learnNorsk-бэкенда

**Диагноз в одну строку:** почти все инциденты (#1 tts-в-10-местах, #2 тумблер грамматики → пустые сессии, #3 audio-pending душит WIP, #4 early_review ворует слуховую партию) — это один и тот же класс бага: **правило существует в N копиях, а не в одной функции с именем**. Значит цель рефактора — не «переложить файлы», а **свести каждое правило к единственной чистой функции с юнит-тестом**, и заставить все каналы её вызывать.

## 1. Целевая структура

```
laereNorskBackand/
├── srs/                        # ЧИСТОЕ ядро. Ноль импортов db.*, ноль SQL, ноль asyncio.
│   ├── cells.py                # RampKind (enum: CONTENT/FUNC_CLOZE/FUNC_CHOICE/PHRASE),
│   │                           #   ramp_kind(pos, data, cloze_enabled) -> RampKind
│   │                           #   required_cells(kind) -> tuple[str, ...]   (кортежи-константы)
│   │                           #   AUDIO_CELL = "choice_no2int"
│   ├── status.py               # word_status(kind, modes, attempts, strength, due_at, now) -> Status
│   │                           #   is_mastered(kind, modes), is_audio_pending(kind, modes, audio_on)
│   ├── steps.py                # next_step(kind, modes, audio_on) -> cell|None
│   │                           #   review_step(kind, modes, audio_on) -> cell|None  ← audio-guard ВНУТРИ
│   ├── schedule.py             # apply_answer(SrsState, correct, elapsed, now) -> SrsState
│   │                           #   (SM-2-шаг, вырезанный из _apply_result_inner; образец — schedule_form)
│   └── gates.py                # compute_wip(words_view) -> WipState(in_work, blocked_new)
│                               #   compute_gate(pack_n, had_cert, throttled) -> GateState(open)
│                               #   ЕДИНСТВЕННАЯ реализация ворот — её вызывают и exams.py, и daily.py
│
├── session/                    # КАНАЛЫ — независимые конвейеры. I/O только в оркестраторах.
│   ├── types.py                # SessionItem (TypedDict): pool_id, mode, direction, step,
│   │                           #   grammar: bool, own_options: bool — тег ЯВНЫЙ, не соглашение по dict-ключу
│   ├── shape.py                # make_element(row, data, mode, direction, step, **extra) -> SessionItem
│   │                           #   единый shape-builder для всех 4 сегодняшних билдеров
│   ├── pools.py                # build_pools(enriched, now) -> CandidatePools(returned, overdue,
│   │                           #   due_mastered, weak, maturing); apply_cooldown(...); dedup_homographs(...)
│   ├── supply.py               # ensure_supply(user_id, ...) — авто-добор слов/фраз (suggest_words),
│   │                           #   ЕДИНСТВЕННОЕ место, где «сборка сессии» пишет в словарь; вызывается ДО build_session
│   ├── forms_phase.py          # plan_forms_phase(FormsInput) -> FormsPlan(picks, phase, save_cycle,
│   │                           #   cap_new, cards_cap, cycle_left, cycle_cells) — чистый конечный автомат words↔forms;
│   │                           #   is_formable(pos, forms) — единый предикат для сессии И note_cycle_mastered
│   ├── distractors.py          # options_patch(items, embcache) -> dict[pool_id, Options] — чистый patch-map,
│   │                           #   не мутирует вход; уважает item["own_options"]
│   ├── reason.py               # empty_reason(session, audio_on, wip, gate, enriched) -> str|None
│   ├── daily.py                # build_session(...) — оркестратор ~150 строк: fetch → pools → gates →
│   │                           #   forms_phase → shape → options_patch → persist(save_cycle)
│   └── listen.py               # build_listen_session / listen_status — свой конвейер поверх srs.status
│
├── db/                         # только доступ к данным
│   ├── learning.py             # ФАСАД: реэкспорт старых имён (build_session, apply_result, ...)
│   │                           #   до конца миграции тестов; реэкспорты — обычными import сверху, не хвостом
│   ├── learning_forms.py       # CRUD form_cycle/form_srs (логика фазы уехала в session/forms_phase.py)
│   ├── exams.py                # канал «экзамен/аудит»; ворота считает через srs.gates
│   └── pool*.py                # + именованные SQL-хелперы: HAS_TTS_SQL, visible_cond(user_id)
│
└── workers/ (autofill*.py, tts.py — на месте, гигиена в этапах 10)
```

**Правила контрактов (нерушимые):**
- `srs/*` и `session/{pools,forms_phase,distractors,reason,shape}.py` — чистые: вход аргументами, выход возвратом. Никаких `get_user_grammar()` изнутри, никаких `runtime.PAUSED`, никакого `await`.
- Профиль юзера читается **один раз** оркестратором в frozen-dataclass `UserPrefs(audio_on, grammar_on, grammar_pos, new_per_session, listen_pack)` и передаётся вниз. Инцидент #2 становится видимым в сигнатуре: `plan_forms_phase(prefs: UserPrefs, ...)`.
- Решения о персисте (`save_cycle`, «добрать слова») чистые функции **возвращают**, а оркестратор **исполняет**. build_session перестаёт быть функцией с сюрпризом «читающее имя — пишет в БД».
- `is`-сравнения с REQUIRED_CELLS/FUNC_CELLS умирают: вместо identity-инварианта — явный `RampKind`.

## 2–3. Этапы strangler (каждый ≤ полдня, деплоится отдельно, поведение не меняется)

Порядок = боль × безопасность. Ни один этап не трогает схему БД ⇒ **откат любого этапа = `git revert` одного коммита + rsync-деплой**, без миграций. Каждый этап заканчивается зелёным `python -m pytest -q` (все 251 существующих + новые).

### Этап 0. Страховочная сеть (до любого переноса кода)

**Цель:** зафиксировать текущее поведение снаружи, где тестов нет вовсе (router-уровень) и где карты нашли дыры.

**Ничего не переезжает.** Только новые тесты:
- `tests/test_api_contracts.py` — первые router-тесты в репо (httpx ASGI-клиент к `main.app`): форма JSON `GET /learning/session` (ключи `words`/`composition`), `GET /learning/listen/status`, `GET /pool/{word}/meta` в обоих исходах (найдено: 10 ключей / не найдено: 2 ключа — фиксируем как есть!), ветвление `POST /learning/answer` (form=true → form_srs, form=false → user_words), `import main` как смок на порядок реэкспортов (ловит ImportError хвоста learning.py:1379+ до прода).
- `tests/test_has_tts_consistency.py` — сид слова → `set_pool_tts` → `hasTts=True` одновременно через `get_pool_list`, `get_pool_meta`, `db.dictionaries.get_user_data`, `db.learning._fetch_user_words`. Это тест, которого не хватало инциденту #1.
- `tests/test_grammar_toggle_session.py` — характеризация дыры #2: ветеран, у которого due только формы, `grammar=False` → сессия НЕ пустая / `reason` корректен (тот самый недостающий тест из карты tests/).
- Golden-тест `build_session`: фиксированный сид → snapshot composition + список (pool_id, mode, step).

**Закрывает:** ничего само по себе — делает все следующие этапы откатываемыми и проверяемыми.

### Этап 3. Бюджеты и ворота: `srs/gates.py`

**Цель:** WIP и ворота экзамена — одна чистая функция, одна на сессию И на экран экзамена.

**Цель:** вынести learning.py:1010–1162 в чистый FSM с явными входами/выходами; убить скрытую мутацию `cap_new` и двойное определение «формо-способности».

## 4. Чего НЕ делать (соблазны = оверкилл для соло-проекта на SQLite)

- **Микросервисы / вынос воркеров в отдельный процесс.** Один инстанс, 23 МБ RAM, asyncio-лупы — норм. Разделение процессов сломает embcache и process-локальные локи ради проблемы, которой нет.
- **Event bus / Celery / Redis / RabbitMQ.** `create_task` + реестр фоновых задач (этап 9) закрывает 100% нужд. Очередь-брокер = новая точка отказа и деплой-сложность.
- **Переезд на PostgreSQL / SQLAlchemy / Alembic.** Идемпотентный DDL в core.py + test_migration.py работают. ORM не решит ни один из четырёх инцидентов — они все про дублирование логики, не про SQL.
- **Нормализация `user_words.modes` в таблицу «для чистоты».** Трогать только если реальная фича потребует; сейчас это перекопает 6+ тестовых файлов с прямым SQL ради нулевой пользы юзеру. (Для форм уже сделано — form_srs — по реальной нужде.)
- **Полная типизация / mypy --strict / переписывание на Pydantic всего внутреннего слоя.** Достаточно dataclass/TypedDict на швах (UserPrefs, SessionItem, FormsPlan, GateState) + response_model на 3–4 роутах.
- **Векторная БД / Redis для эмбеддингов.** Решение уже принято и задокументировано (session-distractors-embcache): RAM-кеш. Не пересматривать.
- **Repository-паттерн / гексагональная архитектура / DI-контейнер.** Слой «чистые функции + тонкие db-аксессоры + оркестратор» — это и есть вся нужная гексагональность без церемоний.
- **API v2 / версионирование контрактов.** Фронт свой, деплоится вместе. response_model + router-тесты дешевле и достаточнее.
- **pytest-xdist / параллельные тесты.** Изоляция держится на подмене глобала `core.DATABASE_URL` — параллелизм даст фантомные гонки. 100 тестов и так быстрые.
- **Погоня за покрытием ради процента.** Тесты — только на контракты швов и инварианты из карт; «дёрнуть каждую строку» не нужно.
- **Чинить найденные картами баги ВНУТРИ рефактор-этапов.** Sev-находки (delete_pool_word сносит оба омонима, leaderboard считает архив «Уже знаю» выученным, орфаны word_tts при переименовании, `LIKE '%uncountable%'` по сырому JSON) — в отдельный баг-бэклог, каждый — отдельный коммит с тестом ПОСЛЕ того, как соответствующий шов вынесен. Смешивание «переноса» и «фикса» — главный способ потерять свойство «этап не меняет поведение».

---

**Итого:** ~7–8 дней чистой работы, 12 независимых деплоев, каждый откатывается одним revert. После этапов 0–3 (два дня) все четыре сегодняшних инцидента становятся структурно невозможными или ловятся юнит-тестом до пуша; этапы 4–8 сводят build_session с ~520 строк до оркестратора, в котором точечная правка канала физически не видит соседей.

# Критика плана (адверсарный проход)

# Адверсарный ревью плана декомпозиции learnNorsk-бэкенда

## Вердикт коротко

План концептуально верный (strangler fig, чистое ядро, факасад-реэкспорт), но в нескольких местах **сам воспроизводит ровно тот класс бага, который призван лечить**: неявные инварианты упорядочивания и синхронизации, зафиксированные в картах только как комментарии, при разбиении на этапы не переносятся в типы/тесты, а «повисают в воздухе» между этапами. Плюс есть один явный самопротиворечие (обещанный `srs/schedule.py` никем не реализуется), недооценка размера самого рискованного этапа и минимум один API-эндпоинт (`/learning/next-cards`), который план унифицирует «бесплатно», хотя карты явно называют его отдельным, потенциально расходящимся контрактом.

## 1. Конкретные правки (этап → правка → почему)

### Этап 0 — страховочная сеть недостаточна

- **Правка:** добавить router-тест на `GET /learning/next-cards` (learning_suggest.next_new_cards) и на `GET /sets/{id}/session` (scoped=True).
  **Почему:** карта api-surface прямо называет `next_new_cards` источником «дрейфа полей между эндпоинтами» (свой dict-литерал, отдельный от `build_session`), а scoped-сессия «полностью выключает WIP/ворота/автодобор одним булевым флагом» без явного контракта. Этап 0 из четырёх перечисленных router-тестов не покрывает ни то, ни другое — а именно эти два пути Этап 6 и Этап 9 трогают сильнее всего.
- **Правка:** golden-тест `build_session` должен (а) сравнивать структурно (parsed dict/JSON), а не строкой, иначе он будет ложно падать на каждом этапе из-за порядка ключей; (б) специально сконструировать входные данные с **связями (tie)** в ключах сортировки (1260/1264), иначе стабильность сортировки — единственный незадокументированный тип-инвариант из карт — никогда не проверится, даже если её сломают.
- **Правка:** тест на «has_tts везде одинаково» должен также покрыть `get_pool_stats`/admin-агрегаты (`/admin/stats`), а не только 4 названные функции — карта explicitly перечисляет 8-10 копий, план тестирует 4.
- **Правка:** тест на грамматический тумблер должен отдельно завести случай «due есть только у грамм-overlay (местоимения/притяжательные)», отдельно — «due есть только у форм-трека». Это два разных механизма (learning_grammar.py vs learning_forms.py), карта прямо разводит их как разные механизмы, а формулировка Этапа 0 («due только формы») покрывает только один.
- **Реализм размера:** «страховочная сеть» — это первая в истории репо router-инфраструктура (httpx ASGI-клиент, вероятно нужна авторизация/JWT для тестов) + golden-снапшот + 3 новых характеризационных файла. Реалистично 1–2 дня, не «полдня» неявно подразумеваемое итоговым расчётом (7-8 дней / 12 этапов).

### Этап 1 — HAS_TTS_SQL, в целом безопасно

- **Правка:** явно включить в тест-план проверку значений `/admin/stats` (два агрегата, по памяти `word-tts-offload-and-stats-perf` они уже были источником рассинхрона) — иначе первая же следующая правка колонки снова тихо расходится именно там, куда план не смотрит.

### Этап 2 — identity-инвариант мигрирован не полностью

- **Правка:** перед переносом сделать **свежий** grep всех `is REQUIRED_CELLS`/`is FUNC_CELLS`/`is not REQUIRED_CELLS` по всему репо (не полагаться на 3 строки, перечисленные в карте — карта фиксирует состояние на момент анализа, а не гарантию полноты).
  **Почему:** это ровно инвариант, который карта называет «молча ломается» при рефакторинге `required_cells` — единственная защита от этого — исчерпывающий grep в момент правки, а план ссылается на конкретные номера строк как на факт.
- **Правка:** тест на паритет со старыми функциями должен включать фикстуру **«старая форма modes до флипа CLOZE_ENABLED»** (grandfathering в `_is_mastered`, строки 188-190), а не только modes, сгенерированные под текущий флаг.
  **Почему:** этот инвариант однонаправленный и НЕ грандфазирует обратный случай (cloze выкл→вкл), карта явно предупреждает — «размастерится без единого ответа». Параметрический генератор, гоняющий только «текущие» modes, эту ветку не заденет.
- **Правка:** удаление `_mastered_by_modes` требует одновременного патча `tests/test_mastery_ramp.py` (единственный потребитель) в том же коммите — иначе `pytest -q` (критерий выхода из этапа) падает не по регрессии, а по ImportError, и это надо явно вписать в объём этапа, а не молчаливо ожидать.

### Этап 3 — дублирующиеся тест-файлы не устранены

- **Правка:** слить/удалить дублирующуюся пару `test_gate_exam.py`/`test_exam_gate.py` в рамках этого же этапа.
  **Почему:** карта explicitly называет их риском («правка ворот требует синхронно править ДВА файла, иначе один остаётся зелёным на устаревшей семантике») — именно момент unify-рефакторинга ворот (`compute_gate`) — точка, где этот риск либо закрывается навсегда, либо тихо переживает рефактор ещё на годы.

### Этап 4 — незаявленное место для early_review

- **Правка:** целевая структура `session/pools.py` не называет early_review-фолбэк явно. Указать прямо, в каком файле он живёт после рефактора (pools.py или daily.py) — это ровно та точка, где случился инцидент #4, и «где именно лежит guard» должно быть частью контракта, а не подразумеваемым.

### Этап 5 — самый рискованный этап недооценён по размеру и по контракту

- **Правка:** явно перенести в контракт `plan_forms_phase` (или в отдельный тип) два инварианта упорядочивания, которые в исходнике держатся только на комментарии на строках 1010-1014:
  1. `ordered_pids` (резервирование) должен быть вычислен **до** построения кандидатов `cands`;
  2. донабор грамм/форм-элементов происходит **после** построения базовой сессии, через множество `in_session` (дедуп «одно слово — один элемент за сессию»).
  **Почему:** ни один из этих двух пунктов не назван в плане нигде явно как требование к сборке в `daily.py` (Этап 8) — они просто «подразумеваются», а именно такое подразумевание карта называет причиной инцидента #2.
- **Правка:** явно перенести инвариант `base_budget = size - len(grammar_picks)` (должен считаться **после** финализации grammar_picks и **до** главного цикла) в тест Этапа 6 или 8 как assert «`len(session) <= size` и не меньше ожидаемого».
  **Почему:** план вообще не упоминает этот инвариант ни разу — это чистый пробел, а нарушение даёт либо переполненную, либо недобранную сессию без единой ошибки.
- **Правка:** добавить `scoped: bool` (или отдельный тип `SessionScope`) в контракт `plan_forms_phase`/`UserPrefs`.
  **Почему:** `UserPrefs` в §1 явно перечисляет только (`audio_on, grammar_on, grammar_pos, new_per_session, listen_pack`) — а `scoped`/`set_id` — второй по значимости «переключатель поведения», который карта называет тем же классом риска («любой новый вызывающий, забывший прокинуть set_id, получает другой набор гейтов без сигнала»). Если план решает именно этот класс бага типизацией префов, странно не закрыть тем же способом второй такой переключатель в тех же самых новых чистых функциях.
- **Размер:** план сам называет этот этап «самая большая боль» / «максимальная» в сводной таблице, но общее правило шапки («каждый ≤ полдня») формально распространяется и на него. Это внутреннее противоречие: конечный автомат с двумя анти-дедлоками, курируемым pronoun-overlay, персистентным циклом и синхронизацией с `note_cycle_mastered` в другом файле — реалистично 2-3 дня, не полдня.

### Этап 6 — «перенос без изменения поведения» здесь на деле меняет контракт

- **Правка:** признать явно, что унификация 4 билдеров в один shape — это **не** poведенчески-нейтральный перенос, а решение об изменении API-поверхности `next_new_cards`/`build_listen_session` (у них сегодня разные поля: `level/freq/topics/hasTts/status/ramp` у next_new_cards против `gloss/example/forms` у build_session). Нужен явный field-by-field diff всех 4 форм **до** написания `make_element`, и решение «только объединение (union), никогда не удаление поля», иначе один из потребителей (включая фронт — «живой добор без пересборки сессии» использует именно next_new_cards) получит урезанный или переименованный набор полей без объявленной причины.
- **Правка:** тест «одинаковый базовый набор ключей» проверяет только ключи, не значения. Добавить тест на **значения** `step`/`mode`/`direction` конкретно для listen-элементов и order/phrase-элементов — по памяти `progress-bar-stage-colors` фронт красит сегмент прогресс-бара по стадии рампы, читая именно эти поля; изменение их семантики при унификации (не имён, а смысла) сломает прогресс-бар молча, ни один из предложенных тестов Этапа 6 этого не поймает.
- **Правка:** явно перечислить в объёме этапа ВСЕ фактические источники элементов, не только `form_element`/`_grammar_element`: main loop (`mode=='study'`, 1183-1193) и main loop `mode=='order'/'cloze'` (1194-1212, устройство и получатель ещё не назван в плане) — иначе Этап 7 получит `KeyError` именно на них (см. ниже).

### Этап 7 — «эквивалентный» рефакторинг фактически ужесточает поведение

- **Правка:** замена `not e.get('grammar')` (безопасный доступ, отсутствие ключа = False) на `not item["own_options"]` (прямая индексация — при отсутствии ключа кидает `KeyError`) — это смена «тихий фолбэк» → «краш», не эквивалентный перенос. Пока не доказано инспекцией **всех** мест построения элемента (включая ещё не мигрированные на Этапе 6 источники — order/cloze), использовать `item.get("own_options", False)` в переходный период, либо явно принять и протестировать факт краша с логированием, какой тип элемента не проставил тег.
  **Почему:** это ровно то самое «новый тип элемента забудет проставить grammar:True», которое карта называет риском — план решает эту проблему типизацией, но одновременно меняет отказоустойчивость с graceful на hard-fail, не заявляя об этом как о сознательном решении.

### Этап 8 — «оркестратор ~150 строк» не содержит явных требований к порядку

- **Правка:** явно вписать в описание Этапа 8 (не полагаться, что «уже вынесенные чистые части» сами сложатся правильно) оба инварианта упорядочивания из Этапа 5-правки выше — сборка `daily.py` из чистых кусков **обязана** вызывать их в конкретном порядке, а нигде в целевой структуре порядок вызовов внутри оркестратора не описан как контракт (только список файлов).
- **Правка:** та жеField-value проверка, что и в Этапе 6, нужна для `build_listen_session` при переносе в `session/listen.py` — это отдельная форма ответа, её тоже неявно «причёсывают» вместе с реэкспортами.

### Этап 9 — переоценённая гарантия и скрытая latency-регрессия

- **Правка:** заявление «двойной тап больше не гоняется за досыпкой и form_cycle» — переоценено. Новый `_SESSION_LOCKS` защищает только путь построения сессии (`ensure_supply + save_cycle`), но **не** пересекается с существующим `_APPLY_LOCK`, которым `apply_result`/`note_cycle_mastered` защищает запись `user_words`/`form_cycle` с другой стороны. Карта прямо называет эту кросс-локовую гонку («save_form_cycle не защищён тем же локом... гонка между двумя вкладками именно на form_cycle»). Нужен **единый** лок-реестр (или единый лок-объект на пользователя), под которым проходят оба пути записи `form_cycle` — иначе Этап 9 закрывает только половину гонки, заявляя, что закрыл её целиком.
- **Правка:** если лок должен охватывать одновременно раннее `ensure_supply` (перед вызовом `build_session`) и позднее `save_cycle` (внутри `build_session`), значит он держится на протяжении **всего** вызова `build_session` — а карта явно фиксирует, что сборка сессии иногда занимает >1с (лог медленной сборки). Это значит: два легитимных параллельных обращения одного юзера (два таба) теперь **сериализуются** на всю длительность сборки, а не только на мутирующий кусок — реальная latency-регрессия, которую предложенный тест («два конкурентных вызова не дублируют досыпку») не измеряет вообще. Либо сузить область лока до двух отдельных коротких критических секций, либо явно замерить и принять деградацию.

### Этап 10 — приоритет боли занижен для известной мины

- **Правка:** `get_pool_id(word)` без `pos` в `complete_batch` (autofill.py:78) — это **известный, документированный** класс инцидента (см. память `autofill-homograph-requeue`: «вечный слив квоты Gemini»), а не гипотетический риск. Вынести этот однострочный фикс из «гигиены» (последний по боли×безопасности этап) в отдельный низкорисковый коммит сразу после Этапа 1 — цена отсрочки (продолжающийся слив квоты/данных на реальных омонимах) выше цены "не по порядку боли".

### Этап 11 — конкретный способ сломать прод response_model'ом

- **Правка:** `response_model=PoolMetaOut` на `/pool/{word}/meta` при двух структурно разных формах ответа (10 ключей найдено / 2 ключа не найдено, которые Этап 0 сам зафиксировал «как есть») почти гарантированно ломает ветку «не найдено»: FastAPI либо отфильтрует лишние поля по модели с недостающими required-полями (422 при сериализации) либо потребует Optional на все 10 полей, ослабляя саму цель «затвердевания». Нужна `Union[PoolMetaFoundOut, PoolMetaNotFoundOut]` или ручная валидация (`response_model=None` + explicit) — иначе Этап 11, единственная цель которого — «расхождение форм становится ошибкой валидации, а не багом», сам вносит настоящую прод-ошибку в первый же деплой.
- **Правка:** то же самое для `SessionOut` на scoped-сессии (`/sets/.../session`) — если `composition` для scoped отличается набором полей (WIP/ворота отключены), единая строгая модель для «обычной» и «scoped» сессии либо потребует Optional-полей везде (снова ослабление цели), либо 422-ит на наборах в проде сразу после деплоя.

### Общее — противоречие по откату и по целевой архитектуре

- **Правка:** формулировку «откат любого этапа = git revert одного коммита» уточнить: это верно только для **последнего** задеплоенного этапа. ## 2. Чек-лист «перед каждым этапом»

1. **Зелёный прогон.** Полный `python -m pytest -q` (все существующие + всё, что добавили предыдущие этапы) + golden/характеризационные тесты Этапа 0 — до первой строки кода нового этапа.
2. **Импорт-смок.** Прогнать `import main` (проверка порядка реэкспортов/транзитивных импортов) **после каждого** этапа, не только после Этапа 0/8 — каждое перемещение функции до полного завершения Этапа 8 рискует сорвать хвостовые импорты `learning_grammar → learning_cloze → leaderboard → placement → exams → learning_suggest`.
3. **Свежий grep, не карта.** Для любого символа с identity-сравнением (`is`) или другим неявным контрактом — искать **все** call-sites заново перед правкой источника; карты фиксируют состояние на момент анализа, не гарантию полноты.
4. **Мутации между вычислением и использованием.** Для каждой переменной, которую этап «замораживает» в параметр чистой функции (frozen dataclass/return value) — явно проверить (grep по записям), нет ли в СТАРОМ коде промежуточной мутации этой переменной между первым вычислением и точками использования, которую новый неизменяемый контракт не воспроизведёт (прецеденты: `cap_new`, `in_work`, `base_budget`, `ordered_pids`/`in_session`).
5. **scoped/set_id — отдельный прогон.** Для каждой новой чистой функции, которая раньше молчаливо меняла поведение через флаг `scoped` внутри `build_session`, — явный тестовый сценарий `scoped=True`, не только «нормальный» путь.
6. **Field-by-field diff перед унификацией форм.** Прежде чем объединять формы ответа/элементов (Этапы 6, 8, 11) — построить точный diff полей и **значений** (не только ключей) старых форм; решение — только объединение (union), никогда не удаление, если хотя бы один потребитель (включая фронт: прогресс-бар по stage-цвету, панели наборов) может читать удаляемое.
7. **`.get()` → `[...]` — осознанное решение.** Любая замена «мягкого» доступа к словарю на «жёсткий» — явная проверка, что ВСЕ производители гарантированно проставляют ключ (инспекция всех мест построения, включая ещё не мигрированные), иначе оставить defensive-get на переходный период.
8. **Локи — не плодить второй реестр.** Любое новое `asyncio.Lock` — проверить, не защищает ли уже существующий лок (в другом файле) ту же сущность; если лок оборачивает вызов целиком (а не узкую критическую секцию) — явно замерить latency-эффект для легитимных параллельных вызовов, не только корректность отсутствия дублей.
9. **Дубли тестов — дедуп сразу.** Если этап трогает логику, покрытую исторически задублированными тестовыми файлами (напр. `test_gate_exam.py`/`test_exam_gate.py`) — слить/удалить дубликат в этом же коммите, не откладывать.
10. **Edge-фикстуры для паритет-тестов.** Параметризованные/сгенерированные тесты «паритет со старым кодом» должны explicitly включать известные особые ветки (grandfathering cloze-toggle, `phase != 'words'` в `note_cycle_mastered`, граница `THROTTLE` строго `>`), а не полагаться на случайную генерацию их нащупать.
11. **Реальная «ветеранская» учётка.** Перед деплоем каждого этапа — прогнать против дампа прод-подобных данных (или заранее заведённого тест-аккаунта уровня `qa_audio`) хотя бы один сценарий с полным набором состояний сразу: due-формы + audio-pending + grammar off + certified/was_certified в обоих направлениях — не только синтетические юнит-фикстуры.
12. **response_model — по всем веткам формы.** Если этап трогает HTTP-контракт — перечислить все структурно разные ветки ответа роута (found/not-found, scoped/unscoped, phase words/forms) и написать по тесту на каждую **до** объявления `response_model`, иначе первый прод-трафик по редкой ветке получит 422/500 вместо ожидаемого ответа.

- _mastered_by_modes (db/learning.py:253-255) — фактически мёртвый код в проде (используется только в tests/test_mastery_ramp.py), при этом имя и сигнатура почти совпадают с канонической _is_mastered(row, modes), но игнорирует required_cells (жёстко предполагает REQUIRED_CELLS) — ловушка для будущей 'простой' замены; стоит либо удалить, либо явно пометить как legacy/test-only.
- routers/learning.py POST /learning/answer — чистая точка ветвления по body.form: явный, легко тестируемый шов между базовой SRS-рампой (мой узел) и треком форм (form_srs); стоит закрепить это тестом на уровне роутера (что form=true никогда не долетает до apply_result и наоборот), а не полагаться только на фронтенд-дисциплину.

## db/learning.py (каналы: build_session, build_listen_session, listen_status, _is_audio_pending, form_cycle: get/save_form_cycle, фаза words↔forms) + db/learning_grammar.py (грамм-overlay ★) + db/learning_forms.py (трек форм — DB-слой form_cycle физически живёт здесь, хотя фигурирует в фокусе задания как часть learning.py)

**Обязанности:**
- build_session (learning.py:770-1286) — единственная точка сборки дневной сессии: приоритезирует пулы (returned→overdue→due_mastered→weak→maturing), берёт для каждого слова next_step, режет по WIP_LIMIT/cap_new/воротам экзамена, ПОТОМ примешивает грамм-квоту (grammar_picks: форма-трек + pronoun-overlay), cloze и order-игры, сортирует (формы вперёд в фазе forms, карточки новых — в конец)
- _is_audio_pending / listen_status / build_listen_session (learning.py:1290-1356) — отдельный слуховой канал: слово, прошедшее 3 текстовые клетки REQUIRED_CELLS кроме choice_no2int (только при gamePrefs.audio=ВКЛ), «ждёт слух»; выдаётся отдельным эндпоинтом /learning/listen, не через /learning/session
- form_cycle (get_form_cycle/save_form_cycle/note_cycle_mastered в learning_forms.py, вызываются из build_session и apply_result) — глобальный ПО ПОЛЬЗОВАТЕЛЮ (не по слову) конечный автомат «words↔forms»: копит выученные формо-способные слова в batch (FORM_CYCLE_BATCH=10), переключает фазу на forms, держит её, пока не сданы ВСЕ клетки партии (produce+interval≥1), затем флипает обратно, плюс два анти-дедлок фолбэка (досрочный флип words→forms, досрочные text-повторы) для случаев, когда основной поток пуст
- learning_grammar.py (_grammar_cells/_grammar_element) — тир ★ overlay НАД уже выученными словами; после выноса форм в learning_forms.py реально обслуживает только pronoun/determiner (курируемая парадигма PRONOUN_PARADIGM), noun/verb/adjective теперь идут через form-трек
- apply_result (learning.py:338-483) — единая точка записи ответа; диспетчерит клетку по принадлежности (base cells vs overlay gcells vs чужая form-клетка) и решает, что можно трогать (due_at/ease/interval/reps слова), а что нельзя — единственное место, которое обязано знать про все три канала одновременно

**Связи и риски:**
- **user_words.modes (JSON) — общий для base-рампы и грамм-overlay** — apply_result пишет ОДНУ и ту же колонку modes и для базовых клеток (choice_int2no, build_int2no…) и для overlay-клеток (choice_gender, input_indefpl, input_objcase…) — ключи различаются только по имени 'mode_direction'; required_cells(row) и _grammar_cells(...) — это ДВА разных набора имён клеток, которые оба читаются из ОДНОГО modes-словаря без namespace-разделения (learning.py:364-366, 383) *Риск: новая грамм-клетка/трек-клетка, чьё имя случайно совпадёт с базовой (или наоборот — сейчас разделены руками через _DIR_ALLOWED и явные списки), тихо перезапишет прогресс рампы или, наоборот, апдейт застрянет в 'защите base-SRS' (learning.py:397-408) — эта защита работает по признаку 'direction not in (no2int, int2no)', т.е. НОВОЕ направление формы обязано не совпадать с 'no2int'/'int2no', иначе проскочит мимо защиты как обычный base-ответ*

- **get_user_audio / get_user_grammar / get_user_grammar_pos (db/users.py, gamePrefs.*)** — build_session читает 3 независимых профильных тумблера (audio_on, grammar_on, grammar_pos) В НАЧАЛЕ функции (learning.py:787-788, 1015-1016) и дальше ветвит ВСЮ логику по ним; audio_on используется в 4 разных местах (_load/in_work, _next_step, early_review-фолбэк, build_listen_session), grammar_on гейтит целиком блок form_cycle (строка 1019 'if grammar_on and not scoped') *Риск: выключение gamePrefs.grammar убирает не только грамм-задания, но и АНТИ-ДЕДЛОК ФЛИП words→forms (learning.py:1087-1100) — единственный механизм, который спасает сессию, если у ветерана все слова mastered и due только у форм; при grammar_on=False эта due-работа просто невидима, сессия отдаёт comp.reason='all_done', хотя реально есть что повторить — воспроизводит инцидент (2) из системного промпта*

- **_review_step / early_review-фолбэк vs слуховой канал** — early_review (learning.py:948-960) вызывает _review_step(row), которая безусловно возвращает 'последнюю непустую НЕ-аудио клетку рампы' (learning.py:281-289) — этот вызов не проверяет modes, просто берёт cells[-1] после фильтрации _AUDIO_CELL; единственное, что удерживает audio-pending слова вне early-фолбэка — явный фильтр 'not (audio_on and _is_audio_pending(...))' в самом списке-кандидате early (строка 954) *Риск: если появится ЕЩЁ один fallback-путь по образцу early_review (а паттерн 'ordered пуст → досыпь из другого пула' уже применялся дважды: анти-дедлок форм 1a889f9 и анти-тупик текста 3bf1975), и он забудет продублировать фильтр audio-pending, то досрочные повторы снова утащат слова из слуховой партии — это и есть инцидент (4) из системного промпта, зафиксированный тем же коммитом 3bf1975, что чинил инцидент (3)*
- **word_pool.forms / word_pool.data (part_of_speech) — общий источник для 3 потребителей** — required_cells (base), _grammar_cells (overlay), form_cells_for (трек форм) — все три параллельно вызывают normalize_pos(data.part_of_speech) и парсят forms JSON независимо друг от друга (learning.py:163-179, learning_grammar.py:115-143, learning_forms.py:93-109), у каждого своя копия логики 'что считать нерегулярным/доступным' *Риск: смена контракта word_pool.forms (например, при следующей чистке/ре-заливке из ordbank, см. ordbank-source-of-truth) обязана быть согласована сразу в трёх местах; поле 'uncountable' форм трогает только form_cells_for — overlay и base о нём не знают, так что несогласованное добавление новых форм-полей рискует тихо разойтись по каналам*
- **db.learning.has_tts EXISTS-подзапрос к word_tts (перенос TTS-колонки)** — _fetch_user_words (learning.py:589) — общий SELECT для ВСЕХ трёх каналов (используется и build_session, и build_listen_session косвенно через _audio_pending_rows, и get_learning/get_due) — джойнит word_tts через EXISTS по norwegian *Риск: это ровно то место, которое задело перенос tts-колонки (commit 401111c) — единая точка чтения слов для каналов означает, что схемные миграции word_pool/word_tts бьют по всем трём каналам разом, а не по одному*

**Инварианты:**
- required_cells(row) содержит РОВНО choice_no2int как _AUDIO_CELL на 2-й позиции (REQUIRED_CELLS) — это жёстко зашито и в _next_step, и в _is_audio_pending, и в _review_step; если порядок клеток в REQUIRED_CELLS когда-нибудь поменяется, _is_audio_pending (проверяет 'все КРОМЕ _AUDIO_CELL == 1') продолжит работать, но комментарии/тесты про 'клетка №2' устареют
- 'mastered' (флаг user_words.mastered) выставляется/снимается ТОЛЬКО в apply_result по _is_mastered(wrow, modes) — это единственное место; form_cycle.note_cycle_mastered вызывается ИСКЛЮЧИТЕЛЬНО на переходе 0→1 (learning.py:464: 'not (st.get("mastered") or 0)'), т.е. предполагается, что apply_result — единственный писатель mastered; set_status(action='know'/'reset') тоже трогает mastered напрямую (учено-подтверждение), но НЕ вызывает note_cycle_mastered — слово, отправленное в архив через 'Уже знаю', не попадает в form_cycle batch, даже если формо-способно
- audio-pending слово ВСЕГДА имеет status in ('learning','review') с due-независимым состоянием — предполагается, что _next_step вернёт None для него естественным путём (skip choice_no2int + все остальные клетки==1); это НЕ проверяется явно нигде кроме комментария (learning.py:262-264) — если когда-то появится способ обойти это (например, откат choice_no2int сбрасывает соседнюю клетку — но код (learning.py:418-423) явно НЕ откатывает при cell==_AUDIO_CELL), инвариант держится только благодаря этому единственному 'if' в apply_result
- form_cycle.phase ∈ {'words','forms'} и form_cycle.batch — это список pool_id КОНКРЕТНО тех слов, что уже mastered=1 на момент попадания в batch; get_form_cycle может вернуть None только 'ни разу не создавалось' — build_session трактует None как 'сид ветерана' (бэклог≥10 → сразу forms), что означает: ПЕРВЫЙ вызов build_session после релиза фичи форм для существующего юзера имеет особый, непереиспользуемый путь инициализации (learning.py:1069-1076)
- grammar_picks (form+overlay) добавляются в session ТОЛЬКО если e['row']['pool_id'] not in in_session (learning.py:1244) — т.е. неявный invariant 'одно слово — максимум один элемент за сессию', который распространяется и на грамм-клетки: если слово уже попало обычным упражнением (например due_mastered-повтором), его форм-клетка в ЭТОЙ сессии молча теряется (не переносится, не откладывается явно) — просто не появится, cycle_left/cycle_cells её всё равно посчитают как 'несданную', создавая расхождение между composition.formsCellsLeft и реально показанными карточками
- apply_result полагается на порядок проверок: cell in gcells (overlay) проверяется РАНЬШЕ, чем cell in cells (base) — 'if cell and cell in gcells and cell not in cells' (learning.py:383) — т.е. если когда-нибудь клетка окажется одновременно и в gcells, и в cells (сейчас невозможно по построению множеств клеток, но не enforced типами), выиграет base-ветка, а overlay ветка тихо не сработает
- цикл words↔forms предполагает единственного текущего исполнителя цикла на пользователя: form_cycle — PRIMARY KEY(user_id), одна строка на юзера; при параллельных вкладках/устройствах (два одновременных /learning/session) оба читают одну фазу, но пишут по своим локальным batch-решениям без блокировки — apply_form_result/apply_result используют лишь общий _APPLY_LOCK (asyncio.Lock, learning.py:335) для user_words, а save_form_cycle не защищён тем же локом, т.е. гонка возможна между двумя вкладками именно на form_cycle

**Горячие точки:**
- learning.py:770-1286 (build_session, целиком) — сходятся ВСЕ каналы разом: base-рампа, слуховой гейт (audio_on/_is_audio_pending), грамм-overlay+форм-трек (form_cycle/grammar_pos), cloze, order-игры (фразы), auto-добор новых слов, ворота экзамена; это самая насыщенная функция кода — почти любое изменение в любом из соседних модулей (users.py тумблеры, learning_forms.py квоты, exams.py ворота) требует правки именно здесь

Одна ошибка в диспетчинге 'какая это клетка' ломает сразу несколько треков одновременно (см. защитные if на строках 383 и 400)

**Швы для разреза:**
- _is_audio_pending(row, modes, data) (learning.py:1290-1298) — чистая функция без БД, чистый вход/выход; уже единственный источник истины для 'слово ждёт слух' — хороший шов: если нужно менять критерий (например добавить фразы в слуховой канал), достаточно править только её и её вызовы (их 5: build_session×2, listen_status, build_listen_session косвенно через _audio_pending_rows, apply_result — нет, apply_result её не вызывает)
- schedule_form(stage, ease, interval_days, correct, elapsed) (learning_forms.py:170-197) — чистая функция, явно described как 'Чистый шаг планировщика' — хороший образец для остального SRS-кода (apply_result/apply_result_inner смешивают чтение БД и SM-2-математику в одном теле, не выделены так чисто)
- form_cells_for/cell_value/form_options (learning_forms.py:80-120) — чистые, принимают pos+forms+cell, не трогают БД; можно тестировать/менять независимо от цикла форм и от build_session
- get_form_cycle/save_form_cycle (learning_forms.py:234-249) — тонкие обёртки над _cycle_row/_save_cycle с явными входом/выходом ({phase,batch} или None) — граница между 'состояние цикла' и 'логика, что с ним делать' (вся логика — в build_session) достаточно чистая, чтобы вынести конечный автомат фазы в отдельную функцию build_session_forms_phase(...), принимающую info/fstates/batch и возвращающую новую фазу+grammar_picks, не трогая остальной build_session
- _grammar_cells/_grammar_element (learning_grammar.py) уже вынесены в отдельный файл с чистым контрактом (data в data-out), реэкспортируются в конец learning.py — модель для дальнейшего дробления build_session: секцию form_cycle (строки 1010-1163) стоит по тому же образцу вынести в learning_forms.py как chosen_form_picks(...), а build_session — только вызывать её

## workers (autofill.py, autofill_enrich.py, autofill_wordgen.py, tts.py) — фоновые LLM/TTS-очереди пула слов

**Обязанности:**
- autofill.py — оркестратор ~10 независимых бесконечных фоновых циклов (create_task в main.py:113-136), каждый — своя очередь по NULL/0-колонке word_pool: describe_loop (description IS NULL, autofill.py:145-163), translate_loop (translate_done=0, autofill.py:174-221), yo_fix_loop (yo_done=0, autofill.py:229-265), reembed_loop (emb_sem=0, батчами до 100, autofill.py:273-310), dedup_loop (dedup_done=0, autofill.py:342-423), freq_loop (freq IS NULL, без LLM, autofill.py:449-471), pos_loop (pos not in POS_KEYS, autofill.py:486-523), forms_loop (forms IS NULL по категории POS, autofill.py:533-566), countability_loop (noun forms без uncountable, autofill.py:579-614)

- complete_batch (autofill.py:65-113) — общая функция 'доделать пачку': эмбеддинг (по pid, разделяя омонимы) + TTS (по написанию, edge, бесплатно) за один проход

'фон работает только в простое' — правило ЛОЖНОЕ для большинства лупов, они молотят постоянно, ограничены только собственным CHECK_SEC и квотой*

**Инварианты:**
- Пул = (norwegian, pos) — ключ пары, а не просто написание (db/pool_homographs...). get_pool_id(word) БЕЗ pos возвращает СТАРШУЮ (ORDER BY id LIMIT 1) запись омонима, а не искомую (db/pool.py:261-277). Все ЦИКЛОВЫЕ выборки (pos_missing_forms, translate_pending, yo_pending, dedup_pending, pool_missing_description/meta) уже возвращают id вместе со словом и пишут строго по нему — это ЗАКОН, введённый после инцидента 24.06 (память autofill-homograph-requeue). Единственное оставшееся нарушение: complete_batch (autofill.py:73-80) — ветка для 'элементов-строк' (не кортежей) всё ещё делает `pid = await get_pool_id(it)` без pos; сейчас безвредно на практике (эмбеддинг для омонима подхватывают reembed_loop/pool_missing_embedding по id), но это тот же паттерн-мина, просто пока не выстрелившая — трогать этот код нужно осторожно, воспроизводя pid-first подход остальных очередей.
- TTS хранится и адресуется ПО НАПИСАНИЮ (word_tts.word PK), а НЕ по (norwegian,pos) — set_pool_tts обновляет клип для ВСЕХ омонимов разом (db/pool_queues.py:23-32, комментарий в памяти autofill-homograph-requeue: «tts проблемы НЕ имеет»). Значит любой код, добавляющий per-pos озвучку (например, разное произношение для разных значений), сломает это допущение — сейчас оно намеренное и задокументировано.
- Порядок в autofill_loop: TTS ПЕРЕД embedding ПЕРЕД classify (autofill.py:644-652, коммит 8df8cfc «озвучка вперёд эмбеддингов»). Инвариант неявный: если эмбеддинг-квота выжата (429), пачка не должна морить бесплатную (edge) озвучку голодом — обеспечивается тем, что `words = tts_miss or emb_miss` берёт TTS-хвост первым.
- Ровно ОДИН активный 'пере-эмбеддинг' механизм за раз: пока идёт reembed_loop (батчами по 100), autofill_loop НЕ добирает эмбеддинги по одному — гейт `emb_miss = ... if (emb_ok and not await sem_embed_pending(1)) else []` (autofill.py:637). Нарушение (например, кто-то уберёт эту проверку) вернёт 'по одному запросу на слово' поверх батчей — сожжёт RPM/дневной лимит запросов.
- pos_loop должен логически идти ПЕРЕД forms_loop («форм без верной POS нет», autofill.py:488) — но это НЕ enforced порядком выполнения (оба — независимые бесконечные циклы, стартуют одновременно в main.py); инвариант держится только тем, что pos_missing_forms(cat,...) фильтрует по конкретной категории POS, так что слово без корректного pos физически не попадёт в выборку ни одной категории, пока pos_loop его не разметит.
- `_NOUN_NOGENDER_TRIES` (autofill_enrich.py:174-215) — счётчик ретраев в памяти ПРОЦЕССА, не в БД: анти-залип (CAP=3) для noun без gender. Инвариант: 'после CAP неудач слово сохраняется БЕЗ рода и больше не в очереди' — но счётчик обнуляется рестартом процесса, а запись может исчезнуть из dict (без очистки) при мёрдже дедупом/сплите омонимов, оставляя мёртвый ключ (мелкая, не растущая до проблемы утечка).
- nouns_missing_countability / countability_progress (db/pool.py:882-909) matching через `forms LIKE '%uncountable%'` / `'%"uncountable": true%'` — строковый поиск по СЫРОМУ JSON-тексту колонки forms, а не JSON-функции SQLite. Инвариант: ключ ДОЛЖЕН называться буквально 'uncountable', а json.dumps ДОЛЖЕН сохранять дефолтные разделители (': ' с пробелом) — переименование поля или смена сериализации в set_pool_forms/merge_pool_forms тихо сломает и очередь, и админский прогресс-бар без единой ошибки в логах.
- Активность (activity.seconds_idle) — это НЕ 'простой всего бэкенда', а 'простой по юзерским LLM-вызовам' (activity.py комментарий). Только autofill_loop проверяет её; остальные 8 очередей крутятся с фиксированным CHECK_SEC независимо от активности юзеров — расхожее ожидание 'фон работает только ночью/в простое' верно ТОЛЬКО для autofill_loop.
- runtime.PAUSED — персистентный (app_settings key='paused') глобальный словарь; ключи фиксированы в runtime.py и НЕ 1:1 с лупами: 'forms' управляет ДВУМЯ разными очередями (forms_loop и countability_loop).

**Горячие точки:**
- _tts_lock (tts.py:20) — сходятся 5 источников: complete_batch (autofill.py), tts_translation_loop (tts.py), GET /tts норвежское слово (routers/pool.py:78), GET /tts?lang= перевод (routers/pool.py:39), POST /pool/{word}/revoice (routers/pool.py:106). Единая точка serialization edge-tts на весь процесс — любой форсаж фоновых констант (TTS_TR_BATCH/CHECK_SEC) напрямую бьёт задержку живого воспроизведения (уже случалось, см. память word-tts-offload-and-stats-perf).
- llm.text_available('autofill') / quota._rr_cursor+_blocked (llm/quota.py) — сходятся ВСЕ 8 фоновых LLM-циклов + классификация в autofill_loop + (частично, через общий fallback-ключ/модель) живые юзерские генерации в autofill_wordgen.py. Общая точка исчерпания квоты без каких-либо приоритетов между воркерами.
- get_pool_id(word[, pos]) (db/pool.py:261) — единственная функция-развилка «правильно по id» vs «наугад по старшей записи»; уже один раз (24.06) ломала описания/классификацию/эмбеддинги при омонимах, остаточный вызов без pos жив в complete_batch (autofill.py:78).
- main.py:113-136 — единая точка старта: все 9 фоновых задач (кроме homograph/tts_tr) висят на ОДНОМ `if text_enabled():`, включая freq_loop, который по документации 'без LLM/сети' (autofill.py:452) — при отсутствии текстовых ключей freq_loop и yo_fix_loop тоже не стартуют, хотя freq не должен зависеть от LLM вообще.
- runtime.PAUSED словарь + POST /admin/control/{key} (routers/pool.py:219-227) — универсальная нетипизированная точка входа: один и тот же переключатель применяется к семантически разным очередям (forms ↔ countability), лёгкий источник 'починил одно — сломал соседнее' при появлении новых лупов без своего флага.
- word_pool как строка: dedup_loop (merge_pool_words, db/pool_dedup.py:72-98) физически УДАЛЯЕТ строку loser, пока describe_loop/translate_loop/pos_loop/forms_loop/countability_loop могут держать в памяти ту же pid между чтением батча и записью результата (await ask_json может занять секунды); UPDATE ... WHERE id=? на уже удалённый id молча matches 0 строк — результат LLM-вызова тихо теряется, квота потрачена впустую. Ни одной блокировки/версионирования между дедупом и остальными 7 очередями нет.

**Швы для разреза:**
- autofill_enrich.py — уже ЧИСТЫЙ слой: classify_batch/describe_batch/translate_batch/forms_batch принимают готовые данные (list[dict] / list[tuple]), возвращают счётчик или dict, не трогают циклы/паузы/активность. Хорошо протестированы изолированно (tests/test_autofill.py:66-119) — безопасное место резать/переиспользовать логику батчей отдельно от расписания.
- autofill_wordgen.py — независим от autofill.py (комментарий в шапке файла: «Зависит только от db/llm/task (не от autofill)») и от runtime PAUSED-семантики циклов, кроме одной точки (_embed_new проверяет runtime.PAUSED['embed'], autofill_wordgen.py:105) — чистый вход/выход (items → pool_id[]), можно тестировать и менять без риска для фоновых лупов.
- complete_batch (autofill.py:65-113) стоило бы разрезать на tts-часть и embed-часть — сейчас это одна функция с двумя разными политиками отказа (tts best-effort/no-quota vs embed quota-gated) и смешанным контрактом входа (кортеж ИЛИ голая строка); разделение убрало бы единственный оставшийся небезопасный get_pool_id(word)-без-pos вызов и сделало бы обе половины независимо тестируемыми.
- langs.py как реестр (LANG_CODES/TTS_VOICE_MAP/TTS_TR_PAIRS) — чистый seam для добавления языка озвучки: одна запись в LANGUAGES, tts.py и routers/pool.py подхватывают её без изменений кода.
- *_loop-обёртки (describe_loop, translate_loop, …) — тонкие, но НЕ протестированы напрямую (только батч-функции внутри них); явный вход/выход у них: db-выборка → батч-функция → db-запись пометки. Если понадобится тестировать таймауты/паузы/idle-логику, это отдельный неприкрытый тестами слой — стоит писать новые тесты именно вокруг циклов, а не только вокруг батчей.
- runtime.PAUSED + persist() (runtime.py) — механически чистый seam (dict + JSON в app_settings), но НЕ типизирован по компонентам; разумный рефакторинг — завести отдельный флаг для countability (сейчас алиас на 'forms') прежде чем добавлять новые лупы, чтобы не плодить дальше one-flag-many-queues.

## pool-data (db/pool.py, db/pool_queues.py, db/ordbank.py, db/core.py — схема word_pool и сателлитов)

**Обязанности:**
- db/core.py: единственный владелец схемы — идемпотентные CREATE TABLE + best-effort ALTER TABLE ADD COLUMN (глотает 'duplicate column') для word_pool и 15+ сателлитных таблиц (word_tts, word_topics, ordbank_ext, nb_lexicon, query_cache, usage, app_settings, dictionaries, dict_words, user_words, push_subscriptions, user_activity, changelog, form_srs, form_cycle, cloze_cache); пул соединений _conn/_release, которым пользуются ВСЕ db/*.py; жизненный цикл ANN-индекса sqlite-vec (vec_words) + brute-force фолбэк; generic KV app_settings.
- db/pool.py: главный CRUD/поиск-слой над word_pool — резолюция идентичности (norwegian,pos), омоним-aware создание (get_or_create_pool + _lemma_redirect: форма→лемма), пятиступенчатый поиск (substring → forms-редирект → fuzzy(rapidfuzz) → semantic(vec) фолбэки), модерация-aware листинг (approved/created_by видимость), admin-статистика/facets, обслуживание forms/description/pos, каскадное удаление; в хвосте файла реэкспортирует pool_moderation/pool_freq/pool_dedup/pool_queues, чтобы `from db.pool import X`/`from db import X` не менялись после дробления модуля.
- db/pool_queues.py: тонкие read/write аксессоры фоновых очередей для воркеров — word_tts (get/set mp3 BLOB), embedding (set_pool_embedding пишет колонку + двигает vec_words + embcache), пары '*_pending / mark_*_done' для translate_done/emb_sem/tts_tr_done/yo_done.
- db/ordbank.py: read-only обёртка над сайдкар-файлом ordbank.db (Norsk ordbank, CC-BY) — детерминированные формы (lookup) + обратный индекс словоформ (formindex: exact_form/prefix_forms/prefix_lemmas) для 'gikk→gå'; плюс живой фолбэк в ord.uib.no (lookup_online), кэшируемый в таблице ordbank_ext основной БД.

**Связи и риски:**
- **word_tts (learning.py:589, dictionaries.py:305, pool.py:566/629/995/1076/1089/1103/1131)** — денормализация по строке написания (norwegian↔word), без FK; фрагмент `EXISTS(SELECT 1 FROM word_tts t WHERE t.word = word_pool.norwegian)` скопирован дословно в ≥8 SQL-запросов в разных файлах вместо одной функции/вьюхи. *Риск: любое изменение схемы TTS (как сегодняшний вынос колонки) требует правки во всех местах руками — ровно инцидент #1 («10 мест»); переименование слова (update_pool_word/replace_pool_word, pool.py:359,387) не чистит и не переносит старую word_tts-запись — висит сирота, а новое написание временно без озвучки.*
- **word_pool.pos ↔ data.part_of_speech (JSON)** — часть речи хранится в двух местах одновременно; синхронизация ручная — get_or_create_pool (pool.py:235-237) пишет оба при создании, set_pool_pos (832-863) синхронизирует существующую запись, но update_pool_word/replace_pool_word колонку pos не трогают. *Риск: рассинхрон колонки pos и data.part_of_speech: код, фильтрующий по _POS_COL (канон-колонка), и код, читающий data.part_of_speech (грамм-overlay, форма сессии), могут не совпасть без единого источника правды.*
- **идентичность (norwegian,pos) ↔ вызовы get_pool_id(word) без pos** — get_pool_id без pos возвращает «любую запись, старшую по id» (pool.py:270-272); delete_pool_word (947-959) вообще матчит по norwegian БЕЗ pos — `DELETE FROM word_pool WHERE norwegian=?` снесёт разом ОБА омонима, а vec_delete/embcache чистятся только для одного id (первого SELECT). *Риск: на словах-омонимах живые вызовы без pos (routers/learning.py:96, routers/sets.py:86, autofill.py:78, tts.py:48, autofill_wordgen.py:82/124) тихо попадают в чужую (старшую) запись — тот же класс бага, что уже чинили в autofill-homograph-requeue; delete_pool_word может оставить в ANN-индексе висячий вектор второго омонима.*
- **word_pool.forms (JSON без схемы) ↔ learning_grammar.py / learning_forms.py / morphology.py** — forms — непрозрачный blob; имена ключей (gender, indef_pl, def_sg, def_pl, past, perfect, present, neuter, plural, comparative, superlative, uncountable, uninflectable) — негласный контракт между тремя писателями (ordbank.py:_parse_uib, LLM forms-батч в autofill.py, scripts/bank_repair.py) и потребителями вне узла (_grammar_cells/_grammar_element в learning_grammar.py, форм-трек в learning_forms.py, form_srs). *Риск: ни один писатель/читатель не типизирован; несовпадение ключа не роняет код, а тихо даёт пустой список грамм-клеток — вероятный механизм инцидента #2 («пустые сессии» после переключения тумблера грамматики: forms есть, но нужных ключей в нём нет/не те).*
- **FORMABLE_POS/_FORMABLE_SQL (pool.py:1019-1025) ↔ clear_nonformable_forms/clear_all_forms/pos_missing_forms/get_pool_stats + внешние POS-ветки в learning_grammar.py/learning_forms.py** — единственный источник правды — жёстко закодированный кортеж ("noun","verb","adjective"); внешние модули дублируют собственные per-POS ветки для того же набора частей речи (grammar-overlay-noun-slice, forms-track). *Риск: добавление новой formable-части речи (напр. местоимения — уже частично вынесены в overlay по памяти проекта) требует синхронной правки кортежа в pool.py и отдельных веток в 2+ модулях учёбы — легко забыть одно место.*
- ***_pending/mark_*_done очереди (translate_done, emb_sem, tts_tr_done, yo_done, dedup_done) ↔ autofill.py-лупы** — плоские 0/NULL-флаги, опрашиваемые LIMIT-батчами; сами функции в pool_queues.py/pool_dedup.py pos-безопасны (возвращают/принимают id, не norwegian) — но безопасность рушится, если вызывающий код в autofill.py заново резолвит id через get_pool_id(word) без pos вместо использования уже данного id. *Риск: autofill.py:217 маркирует translate_done=1 даже при неудаче («попытка сделана — не зацикливаемся») — если апстрим-ошибка систематическая, слово НАВСЕГДА остаётся без перевода, но выглядит «обработанным».*
- **word_pool.embedding ↔ vec_words (sqlite-vec) ↔ embcache (внешний RAM-модуль)** — set_pool_embedding (pool_queues.py:35-48) пишет 3 места НЕ атомарно: сначала колонку в транзакции db, потом vec_upsert (core.py:612, своя connection/commit), потом embcache.update_vec (in-process dict, best-effort в try/except). *Риск: падение процесса между шагами оставляет embedding NOT NULL без ANN-записи (самолечится при рестарте — _init_vec бэкфиллит по `id NOT IN vec_words`, core.py:575-587) или без embcache-вектора (бэкфилла нет в этом узле — слово выпадает из дистракторов до полной перезагрузки embcache).*
- **delete_pool_word/merge_pool_words (pool_dedup.py:72-98) ↔ _cand_cache и _FUZZY_IDX (модульные глобалы pool.py)** — только delete_pool_word явно вызывает _invalidate_candidates() (968) и embcache.remove_vec; update_pool_word/replace_pool_word/get_or_create_pool этого не делают — полагаются на TTL (_CAND_TTL=30с, _FUZZY_TTL=600с) и на то, что COUNT(*) поменяется (для fuzzy-индекса). *Риск: переименование слова (исправление опечатки через replace_pool_word) не меняет COUNT(*) → fuzzy-поиск отдаёт СТАРОЕ написание до 10 минут; distractor-кандидаты — до 30с; нигде не задокументировано, что переименования eventual-consistent.*
- **ordbank_ext (таблица в основной БД, схема — core.py:215-217) ↔ ordbank.py:lookup_online** — второй, отдельный от word_pool механизм «ещё не знаем» — здесь это JSON '{}' как маркер подтверждённого отсутствия, тогда как в word_pool та же семантика («пока не готово») кодируется через NULL/0-флаги (forms IS NULL, COALESCE(x,0)=0). *Риск: два разных соглашения о «пусто = не готово» vs «пусто = подтверждённо нет» в одной кодовой базе — на общий рефакторинг любой из очередей легко перепутать конвенции.*
- **видимость модерации (approved/created_by) ↔ get_pool_list/get_pool_facets/get_pool_meta** — булево условие `(COALESCE(approved,1)=1 OR created_by=?)` вписано инлайн минимум 3 раза (pool.py:665-669 и 1036-1040, плюс внутри get_pool_facets.base()), а не через общий хелпер; сама семантика approved (0/1/2) и её изменение — в pool_moderation.py. *Риск: введение нового статуса модерации (напр. approved=2 «отклонено, но видно админу») требует правки одного и того же булева условия в нескольких местах pool.py синхронно с pool_moderation.py — легко забыть одну из копий.*

**Инварианты:**
- Идентичность = (norwegian, pos), UNIQUE(norwegian,pos) в core.py:188; pos может быть NULL ИЛИ '' — везде используется COALESCE(pos,''), так что NULL и '' обязаны трактоваться как один и тот же «неразмеченный» бакет.
- `forms IS NOT NULL` означает «формы готовы для показа/грамматики», но НЕ означает «эта часть речи вообще предполагает формы» — non-formable POS штатно держат forms=NULL; наличие непустого forms у не-formable слова — след старого бага, для которого специально существует clear_nonformable_forms (pool.py:807-816).
- `embedding IS NOT NULL` — единственный источник истины «есть вектор» одновременно для трёх систем (dedup_pending, vec_words, embcache); ничего не делает рантайм-сверку embcache↔word_pool кроме бэкфилла vec_words при старте (core.py:_init_vec).
- *_pending/mark_*_done — «был обработан» и «обработка удалась» намеренно смешаны (см. autofill.py:217): столбец=1 означает «попытка была», а не «результат есть» — компромисс в пользу отсутствия бесконечных ретраев ценой тихих постоянных пропусков.
- get_or_create_pool: created_by/approved применяются ТОЛЬКО при первой вставке (см. докстринг pool.py:227-231) — вызов на существующей записи их молча игнорирует; попытка «переодобрить»/сменить автора должна идти строго через pool_moderation.set_word_approval, не через get_or_create_pool.
- Переименования (update_pool_word/replace_pool_word) — eventual consistency: старая word_tts-запись не удаляется, _cand_cache и _FUZZY_IDX не инвалидируются явно (только TTL/смена COUNT) — контракт нигде не написан текстом, выведен из кода.
- `_key_cond`/`_pool_relevance` (pool.py:43-55) быстрый путь работает только для языков из `langs.LANG_SET` (_POOL_LANGS); язык вне списка — молчаливый фолбэк на полный LIKE по всему data-блобу (медленно, но корректно) — добавление нового языка интерфейса без регистрации в langs.py деградирует поиск, не ломает его явно.
- db/ordbank.py:_conn/_lock — процесс-синглтон, открывается один раз и никогда не переоткрывается; замена ordbank.db на диске без рестарта процесса не подхватится (то же для DATABASE_URL в core.py — предполагается неизменным на весь процесс).

**Горячие точки:**
- Фрагмент `EXISTS(SELECT 1 FROM word_tts ...)` — сходятся База (поиск/листинг, pool.py), словарь юзера (dictionaries.py:305), построение сессии (learning.py:589), admin missing-фильтр (pool.py:995) и сам TTS-воркер (pool_queues.py) — 3+ фичи буквально в одной скопированной строке SQL; это и есть инцидент #1.
- Колонка `forms` — сходятся грамм-overlay (learning.py:361-366 → learning_grammar.py), форм-трек SRS (learning_forms.py + таблица form_srs), admin-статистика исчисляемости/рода (get_pool_stats, pool.py:570-604), автокомплит/lemma-redirect (_lemma_redirect LIKE-скан по forms, pool.py:199-224), листинг Базы (hasForms в get_pool_list/get_pool_meta) и офлайн-скрипты (bank_repair.py, apply_levels.py) — 6+ потребителей одного нетипизированного blob'а без версии схемы.
- Кластер «резолюция идентичности»: get_or_create_pool + _lemma_redirect + set_pool_pos + merge_pool_words (pool.py/pool_dedup.py) — через них проходят создание слов (autofill_wordgen.py, llm/words.py, llm/phrases.py), сплит омонимов (homograph_split.py), переразметка POS (autofill.py pos_loop) и дедуп одновременно; баг в любой из четырёх функций даёт дубли/потери слов сразу во всех этих фичах.
- word_pool(id) как FK-якорь для dict_words, user_words, form_srs, cloze_cache + внешние vec_words/embcache — delete_pool_word/merge_pool_words обязаны согласованно почистить все 6 хранилищ; сейчас это сделано только для 4 SQL-таблиц + 1 внешнего кеша (embcache), пятое место (vec_words при множественном delete по omonимам) описано в couplings как пропуск.

**Швы для разреза:**

Минус — рефакторинг, переименовывающий/дробящий один из этих флагов (например, разнесение WIP_LIMIT на per-feature лимиты), одномоментно ломает импорт в десятке файлов сразу, и непонятно с первого взгляда, какие из падений — реальные регрессии, а какие — просто устаревший import.*

WIP-лимит считает только реально 'в работе' (in-progress не-mastered) слова и НЕ должен учитывать audio-pending как занимающие слот (test_search_bank.py:107-132).
- WIP_LIMIT ограничивает приток НОВЫХ (0 попыток) слов, но не 'в работе'-слова, которые уже начаты — при лимите новые не подсыпаются, но начатые из выборки выпадать не должны (test_session_engine.py:96-117).
- Порог тормоза аудита — строго '>' THROTTLE, не '>=' (test_audit.py:342-373 отдельно проверяет границу с комментарием про семантику).
- Дедуп ответов ворот-экзамена — по pool_id, не по количеству присланных ответов: повтор одной верной карточки N раз не эквивалентен N разным верным словам (test_gate_exam.py:141-154 `test_gate_no_pass_by_repeating_one_card`, явно описан как анти-чит-регрессия).
- Демоут (провал ворот/аудита, ошибка на due-mastered) откатывает СРАЗУ на последнюю ступень рампы (input сброшен, ранние клетки помечены пройденными), а не на первую клетку — тест явно проверяет это в трёх местах (test_audit.py:173-176, test_gate_exam.py:129-137, test_due_mastered_review.py:79-83).
- Грамм-overlay (★) и трек форм — read-only относительно base-SRS: ни верный, ни неверный ответ по грамм/форм-клетке не трогает reps/ease/interval_days/due_at/mastered базового слова (test_grammar_overlay.py:209-224, test_learning_forms.py:162-185,433-456). Отдельно закреплён гард на СТАРЫЙ формат бандла без form-флага (mode=choice, direction=форма) — тоже не должен просачиваться в base-SRS (test_learning_forms.py:433-456).
- 'hidden'-словарь (авто-пополнение/старт-набор) — синглтон на юзера: suggest_words/seed_starter переиспользуют один и тот же скрытый словарь, не плодят новые, и он никогда не отдаётся в get_user_data (test_dict_study_flags.py:119-184).
- 'Начатое' слово переживает выключение studying у словаря/набора (grandfather-правило): выключается только приток НОВЫХ, а не уже стартовавшие слова (test_dict_study_flags.py:206-217).
- bcrypt усекает пароль на 72 байтах — тест закрепляет это как ожидаемое поведение, а не баг (test_auth_core.py:20-25).
- conftest.py требует DB_POOL_SIZE=0 именно для чистого завершения процесса pytest (комментарий в conftest.py:4-5) — это не случайная настройка, а условие отсутствия зависших соединений/потоков.

**Горячие точки:**
- build_session (db/learning.py) — точка схождения ВСЕХ фич сразу: гейт/ворота (test_exam_gate.py, test_gate_exam.py), аудит-тормоз (test_audit.py), авто-добор (test_auto_replenish.py), WIP-лимит и пейсинг (test_session_engine.py), слуховая партия (test_listen_session.py, test_search_bank.py), трек форм и фазы words/forms (test_learning_forms.py), грамм-overlay (test_grammar_overlay.py), фразы (test_phrases.py), наборы/scoped drill (test_sets.py) — 10 из 34 тестовых файлов реально гоняют build_session и завязаны на его composition-словарь; любая правка сигнатуры/порядка приоритетов рискует одновременно задеть все 10.
- user_words.modes (JSON-строка «1»/«») — общий носитель состояния для base-рампы, грамм-overlay-клеток и (частично) обратной совместимости старых бандлов; проверяется прямыми SQL-чтениями в 6+ файлах (см. couplings выше) — самое похожее по форме на 'вынос tts-колонки' место, если решат нормализовать.
- db/learning.py как модуль-констант-реестр (PACK_FIRST/WIP_LIMIT/NEW_PER_SESSION/THROTTLE/...) — 10+ тестовых файлов делают `from db.learning import <константа>`; это одновременно и защита (тесты не хардкодят значения), и риск (один рефакторинг именования ломает импорт сразу везде).
- word_tts / has_tts — 7 инлайновых corr-subquery копий в разных модулях (db/pool.py×5, db/dictionaries.py, db/learning.py) без единого теста на согласованность (см. coupling выше) — прямое эхо пункта (1) задачи.
